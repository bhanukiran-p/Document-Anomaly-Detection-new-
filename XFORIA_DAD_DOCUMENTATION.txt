PROJECT OVERVIEW
================================================================================

Project Name: XFORIA Document Anomaly Detection (DAD)
Version: 1.0.0
Purpose: AI-powered fraud detection system for financial documents
Technology Stack: React (Frontend) + Flask (Backend) + Supabase (Database)

DESCRIPTION:
XFORIA DAD is an intelligent document analysis platform that uses Machine 
Learning (ML) and Artificial Intelligence (AI) to detect fraud and anomalies 
in financial documents. The system supports four document types:
1. Checks
2. Paystubs
3. Money Orders
4. Bank Statements

The platform also includes real-time transaction monitoring capabilities for
detecting fraudulent patterns in transaction data.

CORE WORKFLOWS
================================================================================

1. DOCUMENT ANALYSIS WORKFLOW (Check/Paystub/Money Order/Bank Statement)
--------------------------------------------------------------------------

Step 1: DOCUMENT UPLOAD
   User uploads document (PDF/Image) via React frontend
   ↓
   Frontend sends file to Flask API endpoint:
   - POST /api/check/analyze
   - POST /api/paystub/analyze
   - POST /api/money-order/analyze
   - POST /api/bank-statement/analyze

Step 2: DOCUMENT EXTRACTION (OCR)
   Backend extracts data using Mindee API or Google Vision
   ↓
   Extracted fields (varies by document type):
   
   Check:
   - Payer name, Payee name, Amount, Check number
   - Date, Bank name, Routing number, Account number
   
   Paystub:
   - Employee name, Employer name, Pay period
   - Gross pay, Net pay, Deductions, YTD earnings
   
   Money Order:
   - Purchaser name, Payee name, Amount
   - Money order number, Date, Issuer
   
   Bank Statement:
   - Account holder, Bank name, Account number
   - Statement period, Opening/Closing balance
   - Transaction history

Step 3: DATA NORMALIZATION
   Extracted data is normalized and validated
   ↓
   - Convert amounts to float
   - Parse dates to standard format
   - Validate required fields
   - Handle missing data

Step 4: ML FRAUD DETECTION
   Normalized data is analyzed by ML model
   ↓
   ML Model (XGBoost) predicts:
   - Fraud Risk Score (0.0 - 1.0)
   - Model Confidence (0.0 - 1.0)
   - Risk Level (LOW/MEDIUM/HIGH/CRITICAL)
   - Top Anomalies (list of detected issues)

Step 5: AI ANALYSIS (GPT-4)
   ML results + extracted data sent to AI agent
   ↓
   AI Agent analyzes:
   - Customer history (repeat vs new customer)
   - Fraud patterns and indicators
   - Contextual anomalies
   - Risk factors
   
   AI Agent returns:
   - Recommendation (APPROVE/REJECT/ESCALATE)
   - Confidence Score (0.0 - 1.0)
   - Summary (natural language explanation)
   - Key Indicators (list of concerns)
   - Fraud Types (if applicable)
   - Fraud Explanations (detailed reasoning)

Step 6: DATABASE STORAGE
   Complete analysis results stored in Supabase
   ↓
   Tables:
   - checks / paystubs / money_orders / bank_statements
   - documents (master table)
   - customer_history (for repeat detection)

Step 7: RESPONSE TO FRONTEND
   JSON response sent back to React frontend
   ↓
   Frontend displays:
   - Risk score and level
   - AI recommendation
   - Detailed analysis
   - Anomalies and fraud indicators
   - Visual risk indicators


2. INSIGHTS DASHBOARD WORKFLOW
--------------------------------------------------------------------------

Step 1: USER NAVIGATES TO INSIGHTS PAGE
   User clicks "Insights" for document type
   ↓
   Frontend loads insights component:
   - CheckInsights.jsx
   - PaystubInsights.jsx
   - MoneyOrderInsights.jsx
   - BankStatementInsights.jsx

Step 2: DATA FETCHING
   Component automatically fetches data from API
   ↓
   API Endpoints:
   - GET /api/checks/list?date_filter=last_30
   - GET /api/paystubs/list?date_filter=last_30
   - GET /api/money-orders/list?date_filter=last_30
   - GET /api/bank-statements/list?date_filter=last_30&start_date=YYYY-MM-DD&end_date=YYYY-MM-DD

Step 3: FILTERING
   User can apply filters:
   ↓
   - Date filters: All Time, Last 30, Last 60, Last 90, Older, Custom Range
   - Bank filter (for bank statements)
   - Search by name/account

Step 4: DATA PROCESSING
   Frontend processes data for visualization
   ↓
   Calculates:
   - Total documents analyzed
   - Average risk score
   - Recommendation distribution (Approve/Reject/Escalate)
   - Risk distribution by range
   - Fraud type distribution
   - Trends over time

Step 5: VISUALIZATION
   Data displayed in charts and metrics
   ↓
   Chart types:
   - Pie charts (recommendations, fraud types)
   - Bar charts (risk distribution, bank comparison)
   - Line charts (trends over time)
   - Metric cards (totals, averages)


3. REAL-TIME TRANSACTION MONITORING WORKFLOW
--------------------------------------------------------------------------

Step 1: CSV UPLOAD
   User uploads transaction CSV file
   ↓
   POST /api/real-time/analyze

Step 2: CSV PROCESSING
   Backend parses and validates CSV
   ↓
   Expected columns:
   - transaction_id, timestamp, amount, merchant
   - customer_id, location, transaction_type

Step 3: FRAUD DETECTION
   ML model analyzes transactions
   ↓
   Detects:
   - Unusual amounts
   - Suspicious patterns
   - Geographic anomalies
   - Velocity checks
   - Merchant risk

Step 4: INSIGHTS GENERATION
   System generates comprehensive insights
   ↓
   - Risk distribution
   - High-risk transactions
   - Pattern analysis
   - Geographic heatmap

Step 5: AI ANALYSIS
   GPT-4 provides natural language summary
   ↓
   - Overall risk assessment
   - Key findings
   - Recommendations

Step 6: VISUALIZATION
   Frontend displays interactive dashboard
   ↓
   - Transaction list with risk scores
   - Geographic map
   - Time-series analysis
   - Risk metrics

FRAUD DETECTION LOGIC
================================================================================

ML MODEL FEATURES:

Common features across all document types:
- Amount (normalized)
- Date features (day of week, month, year)
- Text length features (name lengths)
- Missing field indicators
- Historical customer behavior

Document-specific features:

Checks:
- Check number validity
- Routing number format
- Amount vs typical check amounts
- Payer/Payee relationship

Paystubs:
- Gross vs Net pay ratio
- Deduction percentages
- YTD consistency
- Pay period duration

Money Orders:
- Amount limits (typically $1000 max)
- Issuer validation
- Sequential number patterns

Bank Statements:
- Balance consistency (opening + transactions = closing)
- Transaction patterns
- Unusual transaction amounts
- Geographic anomalies

AI ANALYSIS RULES:

1. Customer Classification:
   - NEW CUSTOMER: First submission or < 3 total submissions
   - REPEAT CUSTOMER: 3+ submissions

2. Fraud Type Detection:
   - Fabricated Document: Low confidence, missing fields
   - Altered Document: Inconsistent formatting, tampering signs
   - Suspicious Patterns: Unusual amounts, odd timing
   - Balance Violations: Math doesn't add up
   - Unrealistic Proportions: Improbable values
   - Repeat Offender: 2+ high-risk submissions

3. Recommendation Logic:
   
   NEW CUSTOMERS:
   - Risk < 30%: APPROVE (no fraud types shown)
   - Risk 30-70%: ESCALATE (no fraud types shown)
   - Risk > 70%: REJECT (no fraud types shown)
   
   REPEAT CUSTOMERS:
   - Risk < 30%: APPROVE (no fraud types shown)
   - Risk 30-70%: ESCALATE (show fraud types)
   - Risk > 70%: REJECT (show fraud types)

FRAUD TYPES EXPLAINED
================================================================================

1. FABRICATED DOCUMENT
   Description: Completely fake document created from scratch
   Indicators:
   - Very low OCR confidence
   - Missing critical fields
   - Invalid document structure
   - Poor image quality
   - Inconsistent formatting

2. ALTERED LEGITIMATE DOCUMENT
   Description: Real document that has been modified
   Indicators:
   - Font mismatches
   - Inconsistent formatting
   - Date discrepancies
   - Signature irregularities
   - Tampered amounts

3. SUSPICIOUS TRANSACTION PATTERNS
   Description: Unusual transaction behavior
   Indicators:
   - Round number amounts
   - Rapid transaction frequency
   - Odd hours transactions
   - Unusual merchant categories
   - Geographic anomalies

4. BALANCE CONSISTENCY VIOLATION
   Description: Mathematical inconsistencies in balances
   Indicators:
   - Opening + transactions ≠ closing balance
   - Amount mismatches
   - Calculation errors
   - Reconciliation failures

5. UNREALISTIC FINANCIAL PROPORTION
   Description: Improbable financial values
   Indicators:
   - Extremely high amounts
   - Unusual ratios (e.g., net pay > gross pay)
   - Excessive deductions
   - Improbable YTD values

6. REPEAT OFFENDER
   Description: Customer with multiple high-risk submissions
   Indicators:
   - 2+ high-risk documents submitted
   - Pattern of suspicious activity
   - Escalating risk scores
   - Consistent anomalies

TROUBLESHOOTING
================================================================================

COMMON ISSUES:

1. "OPENAI_API_KEY not found"
   Solution: Ensure .env file exists in Backend/ with valid API key

2. "Supabase connection failed"
   Solution: Check SUPABASE_URL and SUPABASE_KEY in .env

3. "Mindee extraction failed"
   Solution: Verify MINDEE_API_KEY or use Google Vision as fallback

4. "CORS error in frontend"
   Solution: Ensure Flask-CORS is installed and configured

5. "Model not found"
   Solution: Train ML models using scripts in Backend/*/ml/train_*.py

6. Frontend can't connect to backend
   Solution: Check proxy setting in Frontend/package.json points to correct port

7. "Custom date range not working"
   Solution: Ensure date format is YYYY-MM-DD and start_date < end_date
